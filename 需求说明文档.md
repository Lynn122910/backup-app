## 1. 项目概述

### 1.1 项目背景

在日常工作和学习中，用户需要定期对重要数据进行备份，以防止因硬件故障、误操作、病毒攻击等原因导致数据丢失。本项目的目标是开发一款运行于 Linux 环境的桌面端数据备份与还原软件，提供文件备份、还原、压缩、加密、定时备份等功能，并具备友好的图形用户界面。

### 1.2 项目目标

设计并实现一款功能完善、稳定可靠、易于使用的数据备份软件，满足以下核心目标：

- **数据安全性**：通过加密和完整性校验保障备份数据的安全
- **存储高效性**：通过压缩和增量备份减少存储空间占用
- **操作便捷性**：通过 Qt 图形界面提供直观的操作体验
- **功能完整性**：覆盖备份、还原、过滤、定时、压缩、加密等完整场景

### 1.3 适用范围

适用于个人用户及小型团队在 Linux 桌面环境下对本地文件系统数据进行备份管理。

## 2. 功能需求

### 2.1 基本功能

#### 2.1.1 数据备份（Basic-BK-001）

**描述**：用户选择一个源目录，软件将该目录下的所有文件备份到指定的目标位置。

**输入**：
- 源目录路径（必填）
- 目标备份路径（必填）
- 备份名称/标识（可选）

**处理流程**：
1. 校验源目录是否存在且可读
2. 校验目标路径是否可写，若不存在则自动创建
3. 递归扫描源目录下所有文件
4. 将文件复制/打包至目标路径
5. 生成备份清单文件（manifest），记录备份元信息

**输出**：
- 目标路径下的备份文件/目录
- 备份清单文件（JSON 格式）

**异常处理**：
- 源目录不存在或不可读 → 提示错误，终止操作
- 目标路径磁盘空间不足 → 提示错误，终止操作
- 文件读取权限不足 → 记录警告日志，跳过该文件并继续

---

#### 2.1.2 数据还原（Basic-RS-001）

**描述**：用户选择一个已有的备份，将其中的文件还原到指定目录。

**输入**：
- 备份文件/备份清单路径（必填）
- 还原目标目录（必填）

**处理流程**：
1. 读取备份清单文件，解析备份内容信息
2. 校验目标目录是否可写
3. 将备份文件中包含的所有文件解包/复制到目标目录
4. 根据备份清单恢复文件属性和元数据（如已记录）

**输出**：
- 目标目录下的还原文件
- 还原操作日志

**异常处理**：
- 备份文件损坏或格式不正确 → 提示错误
- 目标路径磁盘空间不足 → 提示错误
- 还原时发生文件冲突 → 提供覆盖/跳过/重命名选项（通过 GUI 交互）

### 2.2 扩展功能

#### 2.2.1 文件类型支持（EXT-FT-001）

**描述**：支持识别和处理 Linux 文件系统中的特殊文件类型，在备份和还原时进行合理处理。

**支持的文件类型**：

| 类型 | 备份行为 | 还原行为 |
|------|----------|----------|
| 普通文件（regular） | 读取内容并存储 | 写入文件内容 |
| 目录（directory） | 递归进入 | 创建目录 |
| 符号链接（symlink） | 记录链接目标路径 | 重新创建符号链接 |
| 硬链接（hard link） | 记录 inode 信息，避免重复存储 | 重新创建硬链接关系 |
| FIFO（命名管道） | 记录类型和路径名 | 重新创建 FIFO |
| 块设备（block device） | 记录 major/minor 号 | ⚠ 仅记录信息，还原时警告 |
| 字符设备（char device） | 记录 major/minor 号 | ⚠ 仅记录信息，还原时警告 |
| 套接字（socket） | 记录类型和路径名 | ⚠ 仅记录信息，还原时警告 |

> ⚠ 标记：设备文件和套接字通常需要 root 权限创建，且属于系统资源，还原时默认跳过并给出提示。

**验收标准**：
- 包含上述所有类型文件的目录经备份再还原后，各文件类型信息一致
- 符号链接还原后仍指向正确目标
- 设备文件备份后还原时给出明确警告

---

#### 2.2.2 元数据支持（EXT-MD-001）

**描述**：备份和还原文件的元数据（属性信息），确保还原后的文件尽可能与原始文件一致。

**支持的元数据类型**：

| 元数据 | 说明 | 备份方式 |
|--------|------|----------|
| 文件权限（mode） | rwx 权限位 | 记录 `st_mode` 值 |
| 所有者（owner） | UID / 用户名 | 记录 UID 和用户名字符串 |
| 所属组（group） | GID / 组名 | 记录 GID 和组名字符串 |
| 修改时间（mtime） | 最后修改时间 | 记录时间戳（纳秒精度） |
| 访问时间（atime） | 最后访问时间 | 记录时间戳（纳秒精度） |
| 状态变更时间（ctime） | 元数据变更时间 | 记录时间戳 |
| 扩展属性（xattr） | Linux 扩展属性 | 记录键值对列表 |
| ACL | 访问控制列表 | 记录 ACL 条目 |

**验收标准**：
- 还原后文件的权限、时间戳与原始文件一致（ctime 除外，因为还原操作本身会更新 ctime）
- 在有权限的情况下，还原后文件的 owner/group 与原始一致
- 扩展属性在支持的文件系统上正确还原

---

#### 2.2.3 自定义备份筛选（EXT-CF-001）

**描述**：支持用户按多种维度筛选需要备份的文件，实现精确控制备份范围。

**筛选维度**：

| 维度 | 说明 | 配置方式 |
|------|------|----------|
| 路径匹配 | 按 glob 模式匹配路径 | 支持 `**`, `*`, `?` 通配符 |
| 文件名匹配 | 按正则表达式匹配文件名 | POSIX 正则 |
| 文件类型 | 筛选特定类型的文件 | 多选：普通文件/目录/链接/... |
| 修改时间 | 按时间范围筛选 | 起始时间 ~ 结束时间 |
| 文件大小 | 按大小范围筛选 | 最小 ~ 最大（字节） |
| 所有者/组 | 按文件所有者筛选 | 用户名/组名字符串 |

**筛选逻辑**：
- 多个条件之间支持 AND（且）/ OR（或）组合
- 支持 include（包含）和 exclude（排除）两种规则
- exclude 规则优先级高于 include 规则

**验收标准**：
- 用户配置筛选条件后，仅符合条件的文件被备份
- include/exclude 规则按优先级正确处理

#### 2.2.4 增量备份（EXT-IB-001）

**描述**：在已完成一次全量备份的基础上，后续备份仅处理发生变化的文件，减少备份时间和存储空间。

**增量策略**：

##### a) 基于时间戳的增量备份

对比文件的修改时间（mtime）与上次备份时间，仅备份修改时间晚于上次备份的文件。

**算法流程**：
1. 读取上次备份的清单文件，获取备份时间戳和各文件快照
2. 扫描源目录，对比每个文件的 mtime
3. 新增文件（清单中不存在）→ 标记为新增，需备份
4. 修改文件（mtime > 上次备份时间）→ 标记为修改，需备份
5. 未变化文件 → 跳过
6. 删除文件（清单中存在但源目录中不存在）→ 在清单中标记为已删除

##### b) 基于校验和的增量备份

对比文件内容的校验和（SHA-256），仅备份内容发生变化的文件，比时间戳更精确但更耗时。

**算法流程**：
1. 读取上次备份清单，获取各文件校验和
2. 扫描源目录，计算当前各文件校验和
3. 对比校验和，不同的文件标记为需备份
4. 新增和删除文件的处理同时间戳策略

**清单数据结构**：
- 全量备份清单：记录所有文件的完整快照
- 增量备份清单：记录本次变更的文件 + 指向上一个清单的链接

**验收标准**：
- 首次执行全量备份，后续执行增量备份
- 增量备份仅处理变更文件，时间显著短于全量备份
- 通过连续清单可完整还原到任意备份点的状态

#### 2.2.5 压缩与解压（EXT-CP-001）

**描述**：对备份数据进行压缩，减少存储空间占用。

**压缩算法**：

| 算法 | 压缩库 | 特点 |
|------|--------|------|
| zlib (Deflate) | zlib | 通用压缩，速度与压缩率均衡 |
| LZ4 | liblz4 | 极速压缩，适合大文件 |
| Zstandard (zstd) | libzstd | 高压缩率 + 高速度，推荐 |

**设计方案**：
- 定义统一的 `Compressor` 抽象接口
- 各算法实现该接口，支持运行时选择
- 压缩在备份过程中流式进行，避免将整个备份先写入磁盘再压缩
- 备份文件格式：`[压缩头][压缩后的数据块]...`

**验收标准**：
- 备份文件体积明显小于原始文件总体积
- 压缩→解压→还原后，文件内容与原始完全一致
- 支持不同压缩级别（快速/均衡/最大压缩）

---

#### 2.2.6 加密与解密（EXT-EN-001）

**描述**：对备份数据进行加密，保障数据机密性。只有持有正确密码的用户才能还原备份。

**加密算法**：

| 算法 | 加密库 | 特点 |
|------|--------|------|
| AES-256-GCM | OpenSSL | 对称加密 + 认证加密（AEAD），推荐 |
| AES-256-CBC + HMAC | OpenSSL | 对称加密 + 独立完整性校验 |
| ChaCha20-Poly1305 | OpenSSL / libsodium | 替代方案，移动端友好 |

**设计方案**：
- 用户输入密码，通过 PBKDF2 / Argon2 派生加密密钥
- 随机生成 IV / Nonce，存储于备份文件头部
- 采用认证加密（AEAD）模式，同时提供机密性和完整性保护
- 备份文件格式：`[Salt][IV/Nonce][认证标签][加密数据]`

**密钥派生流程**：
```
用户密码 → PBKDF2-HMAC-SHA256(密码, Salt, 100000轮迭代) → 256位密钥
```

**验收标准**：
- 加密备份文件内容不可直接读取
- 输入正确密码可成功解密还原
- 输入错误密码时解密失败并提示
- 加密数据被篡改后解密失败（完整性保护）

#### 2.2.8 定时备份（EXT-TB-001）

**描述**：用户可以配置定时备份策略，软件在指定时间自动执行备份任务。

**定时策略**：

| 策略类型 | 说明 | 配置参数 |
|----------|------|----------|
| 间隔备份 | 每隔固定时间执行 | 间隔时长（分钟/小时） |
| 每日定时 | 每天固定时间执行 | 时间点（HH:MM） |
| 每周定时 | 每周固定日期时间执行 | 星期几 + 时间点 |
| 每月定时 | 每月固定日期执行 | 日期 + 时间点 |
| Cron 表达式 | 任意复杂周期 | cron 表达式字符串 |

**自动淘汰机制**：
- 按数量淘汰：保留最近 N 个备份，自动删除更旧的
- 按时间淘汰：删除超过 N 天的旧备份
- 按空间淘汰：当备份总大小超过阈值时删除最旧的备份
- 智能淘汰：结合以上策略，保留合理数量的历史备份

**实现方式**：
- 使用 Linux cron 或 systemd timer 作为底层定时机制
- 备选：使用 Qt 的 `QTimer` 实现应用内定时器
- 定时任务持久化到配置文件，软件重启后自动恢复

**验收标准**：
- 到达预定时间后自动启动备份任务
- 后台运行时不影响前台操作
- 定时任务配置持久保存
- 自动淘汰按策略正确执行

## 3. 非功能需求

### 3.1 性能需求

| 指标 | 要求 |
|------|------|
| 备份吞吐量 | ≥ 50 MB/s（SSD 环境下） |
| 增量备份速度 | 扫描 10 万文件 ≤ 5 秒 |
| GUI 响应时间 | 用户操作响应 ≤ 200ms |
| 内存占用 | 常态 ≤ 200MB；备份过程中峰值 ≤ 500MB |
| CPU 占用 | 备份过程中平均 ≤ 50%（单核） |

### 3.2 可靠性需求

- 备份过程中发生异常（断电、磁盘满）时，已备份的数据应可用，不应整体损坏
- 支持断点续传：大型备份中断后可从断点继续
- 备份文件自带完整性校验（CRC32 / SHA-256）
- 软件连续运行 72 小时无内存泄漏（通过 valgrind 验证）

### 3.3 安全性需求

- 加密密钥不得明文存储于磁盘
- 密码输入框应掩码显示
- 备份文件完整性受校验码保护
- 不使用已知有漏洞的加密算法（如 DES、RC4）

### 3.4 兼容性需求

- 操作系统：Ubuntu 20.04+、Debian 11+、CentOS 7+、Fedora 36+
- 文件系统：ext4、xfs、btrfs（部分元数据功能因文件系统而异）
- Qt 版本：兼容 Qt 5.15 及 Qt 6.x

### 3.5 可维护性需求

- 代码遵循 Google C++ Style Guide
- 使用 cpplint 进行代码风格检查
- 核心模块单元测试覆盖率 ≥ 70%（使用 gtest）
- 关键接口和数据结构有完整注释（Doxygen 格式）

## 6. 约束与假设

### 6.1 技术约束

- 必须使用 C/C++ 语言实现核心逻辑

- 运行环境限定 Linux

- 备份目标存储介质有足够空间
- 系统已安装 Qt 5.15+ 运行时库
- 对于需要 root 权限的操作（如恢复设备文件、更改文件所有者），软件给出提示而非直接失败
